# 티카투카 스타일 PvP 사이드 프로젝트 — Codex 작업 인수인계

> 작성 목적: 로스트아크의 `티카투카` 및 원형으로 알려진 `Knucklebones` 계열에서 영감을 받은 **별도 사이드 프로젝트**의 MVP를 구현하기 위한 요구사항 정리 문서  
> 주의: 본 프로젝트는 **Artel과 완전히 별개**다. AI Agent, 로그라이크, 프롬프트, Artel 관련 요구사항은 포함하지 않는다.

---

## 1. 프로젝트 한 줄 정의

사용자가 방을 생성하거나 방 코드로 참가해, 친구와 **1:1 턴제 주사위 배치 PvP**를 플레이할 수 있는 웹 게임 MVP.

---

## 2. 개발 목표

MVP의 목표는 아래 한 가지다.

> 두 명의 사용자가 같은 방에 접속하여, 티카투카/Knucklebones 계열의 핵심 규칙으로 한 판을 정상적으로 진행하고 승패를 확인할 수 있어야 한다.

초기 버전에서 랭크, 자동 매칭, 시즌, 재화 등 라이브 서비스 요소는 구현하지 않는다.

---

## 3. MVP 범위

### 포함

- 닉네임 입력
- 방 생성
- 방 코드 생성 및 공유
- 방 코드로 방 참가
- 2명 입장 시 게임 시작
- 1:1 실시간 턴제 게임
- 3x3 보드
- 일반 주사위 기반 배치
- 같은 숫자 중첩에 따른 점수 계산
- 상대 주사위 제거 규칙
- 턴 제한 시간
- 게임 종료 및 승패 표시
- 다시 하기 / 방 나가기

### 제외

- 랭크
- 자동 매칭
- 시즌
- 친구 시스템
- 회원가입 / 로그인
- 전적 저장
- 리플레이
- 관전
- 이모트
- 상점 / 재화 / 결제
- AI 상대
- 실드 주사위
- 리롤
- 스킬형 주사위
- 복잡한 특수 기물
- 방장 강퇴
- 재접속 복구
- 다인전

## 4. 핵심 게임 콘셉트

### 기본 구조

- 플레이어는 각자 `3 x 3` 보드를 가진다.
- 각 턴마다 주사위 1개를 받는다.
- 플레이어는 주사위를 자신의 보드에 있는 빈 칸 중 하나에 배치한다.
- 같은 라인에 같은 숫자를 모아 점수를 높인다.
- 특정 조건에서 상대 보드의 주사위를 제거할 수 있다.
- 보드가 채워지거나 종료 조건을 만족하면 점수를 계산해 승패를 결정한다.

### 게임의 재미 요소

- 높은 숫자를 배치하는 단순 확률 게임이 아니다.
- 같은 숫자를 모아 점수를 증폭시키는 빌드업이 있다.
- 상대가 모은 숫자를 제거하며 흐름을 끊는 상호작용이 있다.
- 3개의 라인 중 어느 라인에 투자하고, 어느 라인을 포기할지 판단하는 전략성이 있다.
- 한 턴의 배치가 내 득점과 상대 방해를 동시에 만들 수 있다.

## 5. 규칙 정리

## 5.1 보드

```text
플레이어 A 보드                 플레이어 B 보드

[ ][ ][ ]                      [ ][ ][ ]
[ ][ ][ ]                      [ ][ ][ ]
[ ][ ][ ]                      [ ][ ][ ]
```

- 각 플레이어는 총 9칸의 보드를 가진다.
- 주사위는 빈 칸에만 배치할 수 있다.
- 이미 주사위가 있는 칸에는 둘 수 없다.
- 한 번 둔 주사위는 기본적으로 이동할 수 없다.

---

## 5.2 턴 진행

기본 턴 흐름은 아래와 같다.

```text
현재 플레이어 턴 시작
→ 주사위 숫자 생성
→ 플레이어가 빈 칸 하나 선택
→ 서버가 배치 가능 여부 검증
→ 주사위 배치
→ 상대 주사위 제거 판정
→ 점수 재계산
→ 종료 조건 확인
→ 다음 플레이어 턴 전환
```

### 턴 제한

- 권장 제한 시간: `10~15초`
- MVP 기본값 제안: `15초`
- 시간 초과 시 처리 방식은 아래 둘 중 하나를 선택한다.

#### 권장안 A: 랜덤 빈 칸 자동 배치
- 현재 주사위를 남은 빈 칸 중 하나에 랜덤 배치한다.
- 게임 흐름이 끊기지 않는다.
- 구현이 단순하다.

#### 대안 B: 자동 패배
- 시간 초과 시 즉시 패배 처리한다.
- 경쟁성은 높지만 MVP 테스트 단계에서는 이탈/불쾌감을 높일 수 있다.

MVP에서는 **A안: 랜덤 빈 칸 자동 배치**를 권장한다.

---

## 5.3 점수 규칙

같은 라인 안에서 같은 숫자를 여러 개 배치할수록 점수가 증가한다.

### 권장 점수 공식

한 라인에서 같은 숫자 `n`개가 있다면:

```text
숫자 값 × n²
```

예시:

| 라인 구성 | 계산 | 점수 |
|---|---:|---:|
| 6 | 6 × 1² | 6 |
| 6, 6 | 6 × 2² | 24 |
| 6, 6, 6 | 6 × 3² | 54 |
| 2, 4, 6 | 2 + 4 + 6 | 12 |
| 3, 3, 5 | (3 × 2²) + 5 | 17 |

이 방식은 Knucklebones 계열에서 널리 쓰이는 직관적인 구조다.

> 구현 시에는 각 행 또는 열을 하나의 `Line`으로 보고, 같은 눈의 개수를 세어 합산하면 된다.

---

## 5.4 상대 주사위 제거 규칙

### 현재 구현 전 반드시 확정해야 하는 항목

이 프로젝트의 가장 중요한 규칙 확정 사항이다.

기존 대화에서는 로스트아크 티카투카의 제거 기준을 두고 `같은 줄`, `같은 열` 표현이 혼재되었다.  
원형 Knucklebones는 보통 **내가 배치한 주사위와 대응되는 상대 보드의 같은 열에서, 같은 숫자 주사위를 제거**하는 방식으로 알려져 있다.

따라서 구현 전 아래 중 하나를 확정해야 한다.

#### 권장안: 같은 열 제거

```text
내 보드                            상대 보드

[ ][5][ ]                          [ ][5][ ]
[ ][ ][ ]          배치 전          [ ][2][ ]
[ ][ ][ ]                          [ ][5][ ]

내 중앙 열에 5를 배치하면
상대 중앙 열의 숫자 5를 모두 제거
```

- 내가 특정 열에 숫자 `X`를 둔다.
- 상대의 **같은 열**에 숫자 `X`가 있으면 해당 숫자를 모두 제거한다.
- 같은 숫자가 여러 개 있으면 모두 제거한다.
- 숫자가 다른 주사위는 유지된다.

이 방식이 원형 게임의 전략성과 가장 잘 맞는다.

#### 대안: 같은 행 제거

- 내가 특정 행에 숫자 `X`를 둔다.
- 상대의 같은 행에서 숫자 `X`를 제거한다.

이 방식도 가능하지만, 원형과 체감이 달라질 수 있다.

### MVP 권장 결론

> **같은 열 제거**를 기본 규칙으로 구현한다.

단, 실제로 로스트아크 티카투카를 그대로 재현하려는 경우에는 인게임 규칙을 별도로 확인한 뒤 상수 또는 규칙 모듈만 바꾸도록 설계한다.

## 5.5 승리 조건

현재 다음 두 가지 후보가 있다.

### 후보 A: 총점 승리

- 게임 종료 후 3개 라인의 점수를 모두 합산한다.
- 총점이 높은 플레이어가 승리한다.
- 동점이면 무승부 또는 타이브레이커를 적용한다.

장점:
- 이해가 쉽다.
- 구현이 가장 단순하다.

### 후보 B: 라인 다수 승리

- 각 라인 점수를 상대의 대응 라인 점수와 비교한다.
- 3개 라인 중 2개 이상 이긴 사람이 승리한다.
- 라인 승수가 같을 때 총점으로 타이브레이커를 한다.

장점:
- 특정 라인을 포기하고 2개 라인에 집중하는 전략이 생긴다.
- 티카투카식 라인 대결 감각을 강화할 수 있다.

### MVP 권장 결론

> MVP는 **총점 승리**를 우선 권장한다.

이유:
- 규칙 설명과 UI가 단순하다.
- 유저 테스트에서 점수 계산을 직관적으로 이해하기 쉽다.
- 이후 원한다면 `라인 다수 승리 모드`를 추가할 수 있다.

---

## 5.6 게임 종료 조건

### 기본 종료

- 한 플레이어의 보드가 9칸 모두 채워지면, 다른 플레이어도 남은 배치를 마칠 수 있도록 처리한다.
- 양쪽 보드가 모두 채워지면 게임을 종료한다.
- 최종 점수를 계산한다.
- 승리 / 패배 / 무승부를 표시한다.

### 권장 상세 처리

- 한쪽 보드가 가득 차면 그 플레이어는 더 이상 배치하지 않는다.
- 상대 플레이어는 자신의 빈 칸을 모두 채울 때까지 턴을 진행한다.
- 양쪽 모두 보드가 가득 찬 시점에 종료한다.

이 방식이 양쪽 플레이어에게 동일한 총 배치 기회를 준다.

## 6. 화면 구성

MVP에서는 화면을 크게 3개로 제한한다.

---

## 6.1 시작 화면

### 목적

- 플레이어 닉네임 입력
- 방 생성 또는 방 참가

### 필요한 UI

```text
[ 게임 로고 ]

닉네임
[____________]

[ 방 만들기 ]

방 코드
[____________]
[ 방 참가 ]
```

### 동작

- 닉네임이 비어 있으면 방 생성/참가 버튼 비활성화 또는 경고 표시
- 방 생성 성공 시 대기방으로 이동
- 방 참가 성공 시 대기방 또는 게임 화면으로 이동
- 잘못된 방 코드 입력 시 오류 메시지 표시

---

## 6.2 대기방

### 목적

- 방 코드 공유
- 상대방 접속 대기
- 2인 입장 상태 확인

### 필요한 UI

```text
방 코드: ABC123
[ 복사 ]

플레이어 1: 김태민      준비 완료
플레이어 2: 참가 대기 중

상대방이 참가하면 게임이 자동으로 시작됩니다.

[ 방 나가기 ]
```

### 동작

- 방 생성 시 6자리 영문/숫자 방 코드 생성
- 두 번째 플레이어가 입장하면 게임을 자동 시작
- 방장만 있는 상태에서는 `상대방 참가 대기` 표시
- 한 명이 나가면 상대에게 퇴장 메시지를 표시하고 대기방 또는 시작 화면으로 복귀

---

## 6.3 게임 화면

### 핵심 UI

```text
상대 닉네임              상대 남은 칸 / 점수
[ 상대 보드 3x3 ]

현재 턴 표시
남은 시간: 15초

현재 주사위
[ 5 ]

[ 내 보드 3x3 ]

내 닉네임                내 남은 칸 / 점수
```

### 필수 정보

- 내 닉네임
- 상대 닉네임
- 내 보드
- 상대 보드
- 현재 주사위 숫자
- 현재 턴 플레이어 표시
- 남은 턴 시간
- 현재 점수
- 최근 이벤트 로그 1~3줄

### 이벤트 로그 예시

```text
김태민이 5를 중앙 열에 배치했습니다.
상대의 5 주사위 2개가 제거되었습니다.
상대 차례입니다.
```

### 종료 모달

```text
[ 승리! ]

내 점수: 42
상대 점수: 36

[ 다시 하기 ]
[ 방 나가기 ]
```

## 7. 방 시스템 요구사항

## 7.1 방 생성

- 사용자가 닉네임을 입력한다.
- 서버가 중복되지 않는 방 코드를 생성한다.
- 방 생성자는 Player 1로 입장한다.
- 방 상태는 `WAITING`으로 설정한다.

### 권장 방 코드

- 길이: 6자리
- 구성: 대문자 알파벳 + 숫자
- 혼동하기 쉬운 문자 제외 권장: `0`, `O`, `1`, `I`, `L`

예시:

```text
AB7K9Q
X4PM82
```

---

## 7.2 방 참가

- 사용자가 닉네임과 방 코드를 입력한다.
- 서버가 방 존재 여부를 확인한다.
- 방이 이미 2명이라면 참가를 거절한다.
- 방에 정상적으로 입장하면 Player 2가 된다.
- 두 명이 입장한 순간 게임을 시작한다.

### 예외 처리

| 상황 | 처리 |
|---|---|
| 존재하지 않는 방 코드 | `존재하지 않는 방입니다.` |
| 이미 게임 중인 방 | `이미 게임이 시작된 방입니다.` |
| 이미 인원이 찬 방 | `방 인원이 가득 찼습니다.` |
| 닉네임 없음 | `닉네임을 입력하세요.` |
| 동일 브라우저 중복 입장 | MVP에서는 별도 방지 없이 허용 가능, 단 테스트 시 주의 |

---

## 7.3 퇴장 처리

MVP 권장 정책:

- 대기방에서 나가면 방을 삭제하거나 빈 방으로 유지한다.
- 게임 중 한 명이 나가면 남은 플레이어를 승리 처리한다.
- 남은 플레이어에게 `상대방이 퇴장했습니다.`를 보여준다.
- 이후 시작 화면으로 돌아간다.

재접속 복구는 MVP에서 제외한다.

---

## 8. 실시간 동기화 요구사항

이 게임은 턴제이지만, 두 플레이어의 보드와 턴 상태를 즉시 맞춰야 한다.

### 권장 방식

- WebSocket 기반 실시간 통신
- REST API는 방 생성/참가 등 초기 요청에 사용 가능
- 실제 게임 상태 변경은 WebSocket 이벤트로 처리

### 서버 권한 원칙

> 클라이언트는 “어느 칸에 두겠다”만 요청하고, 모든 게임 판정은 서버가 수행한다.

서버가 담당해야 하는 항목:

- 현재 턴인지 확인
- 선택한 칸이 비어 있는지 확인
- 주사위 숫자 생성 또는 검증
- 주사위 배치
- 상대 주사위 제거
- 점수 계산
- 종료 조건 판정
- 승패 결정
- 다음 턴 전환
- 타이머 처리

클라이언트가 담당해야 하는 항목:

- UI 표시
- 칸 선택 입력
- 애니메이션
- 사운드
- 서버 이벤트 수신 후 화면 갱신

### 보드 표현 예시

```json
  "board": [
    [null, 5, null],
    [2, 5, null],
    [null, null, 3]
  ]
}
```

### 라인 점수 계산 예시

```json
{
  "lineScores": [10, 22, 3],
  "totalScore": 35
}
```

## 11. 개발 우선순위

### 1단계 — 로컬 게임 규칙

온라인 기능 없이 한 화면에서 아래가 동작해야 한다.

- 2명 번갈아 턴 진행
- 주사위 생성
- 칸 배치
- 같은 숫자 점수 계산
- 상대 주사위 제거
- 게임 종료
- 승패 계산

### 2단계 — 게임 로직 테스트

다음 케이스를 테스트한다.

- 빈 칸에만 배치되는지
- 상대 턴에는 배치할 수 없는지
- 같은 숫자 2개/3개 점수가 맞는지
- 동일 열 제거가 모두 적용되는지
- 제거 후 점수가 즉시 갱신되는지
- 보드가 가득 찬 플레이어의 턴을 건너뛰는지
- 종료 시 승패가 정확한지

### 3단계 — 방 생성 / 참가

- 방 코드 생성
- 방 코드 복사
- 방 코드 참가
- 2명 제한
- 대기방 상태 표시

### 4단계 — WebSocket 실시간 동기화

- 게임 시작 동기화
- 턴 동기화
- 보드 동기화
- 제거 이벤트 동기화
- 승패 동기화

### 5단계 — 타이머 / 예외 처리

- 턴 제한 시간
- 시간 초과 자동 배치
- 게임 중 퇴장
- 잘못된 요청 처리

### 6단계 — UI/연출 정리

- 주사위 굴림 애니메이션
- 배치 애니메이션
- 제거 애니메이션
- 턴 전환 효과
- 결과 모달
- 모바일/데스크톱 기본 반응형 처리

## 12. MVP 완료 기준

아래가 모두 충족되면 MVP 완료로 본다.

- [ ] 플레이어가 닉네임을 입력할 수 있다.
- [ ] 플레이어가 방을 만들 수 있다.
- [ ] 방 코드가 생성되고 복사할 수 있다.
- [ ] 다른 플레이어가 방 코드로 참가할 수 있다.
- [ ] 두 명이 입장하면 게임이 시작된다.
- [ ] 각 플레이어에게 3x3 보드가 표시된다.
- [ ] 현재 턴인 플레이어만 주사위를 둘 수 있다.
- [ ] 서버가 배치를 검증한다.
- [ ] 동일 숫자 중첩 점수가 정상 계산된다.
- [ ] 같은 열의 동일 숫자 주사위가 상대 보드에서 제거된다.
- [ ] 턴 타이머가 동작한다.
- [ ] 시간 초과 시 자동 배치된다.
- [ ] 보드 종료 조건이 정상 처리된다.
- [ ] 최종 점수와 승패가 양쪽 화면에 동일하게 표시된다.
- [ ] 상대가 나가면 남은 플레이어에게 퇴장 안내가 표시된다.

## 13. 구현 전 최종 확인이 필요한 결정 사항

아래 3개는 코드를 본격적으로 작성하기 전에 확정해야 한다.

| 항목 | 권장안 | 이유 |
|---|---|---|
| 주사위 제거 기준 | 같은 열 | Knucklebones 계열의 핵심 감각과 가장 가깝다. |
| 승리 조건 | 총점 승리 | MVP에서 가장 직관적이고 구현이 단순하다. |
| 시간 초과 처리 | 랜덤 빈 칸 자동 배치 | 게임이 중단되지 않고 테스트가 쉽다. |

현재 문서는 위 권장안을 기본값으로 둔다.

## 14. 향후 확장 후보

MVP 이후 우선순위는 아래처럼 잡는다.

```text
v1.1 리롤
v1.2 실드 주사위
v1.3 다시 하기 고도화 / 재접속
v1.4 친구 초대 링크
v1.5 관전 및 리플레이
v1.6 자동 매칭
v1.7 랭크 / 전적 / 시즌
```

### 실드 주사위 확장 아이디어

- 일반 주사위와 달리 제거되지 않는 보호 기물
- 내 보드 보호용 또는 상대 칸 방해용
- MVP에서 제외한 이유: 규칙 설명, 밸런스, UI 상태가 복잡해지기 때문

### 리롤 확장 아이디어

- 턴당 또는 게임당 제한 횟수의 리롤 제공
- 현재 숫자와 새 숫자 중 하나를 선택
- 단순 랜덤성 완화 및 역전 요소 제공

## 15. Codex 작업 지시 요약

1. 먼저 로컬 환경에서 `GameEngine` 또는 `GameService`를 만들어 게임 규칙을 독립적으로 구현한다.
2. 게임 규칙은 UI와 분리하고, 입력 상태 → 결과 상태를 반환하는 순수 함수에 가깝게 설계한다.
3. 서버가 모든 판정을 갖고, 프론트엔드는 표시와 입력만 맡는다.
4. 이후 방 생성/참가와 WebSocket 동기화를 붙인다.
5. UI는 화려함보다 게임 상태가 명확하게 보이도록 먼저 만든다.
6. MVP에서는 특수 주사위, 리롤, 랭크, 매칭을 절대 먼저 넣지 않는다.

## 16. 핵심 원칙

```text
기본 규칙이 재미있어야 특수 규칙도 재미있다.
```

따라서 첫 목표는 아래 경험을 완성하는 것이다.

```text
방 생성
→ 친구 초대
→ 1:1 접속
→ 주사위 배치
→ 상대 주사위 제거
→ 점수 역전
→ 승패 확인
```

이 한 사이클이 매끄럽게 동작하면 MVP 성공이다.
